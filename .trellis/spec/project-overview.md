# AgentDashboard - 项目总览

> **本文件是所有spec文档的核心锚点，开始任何开发工作前必须阅读。**

---

## 项目定位

AgentDashboard 是一个**统一看板系统**，用于在多设备、多项目中管理 AI Agent 的协同工作。

**核心灵感：**
- 胡渊鸣实践：多个 Claude Code 实例并行工作，通过任务队列循环（Ralph Loop）驱动
- vibe-kanban：以工作空间为中心管理 coding agent 的看板
- Trellis：提供 Hook 注入机制和 Multi-Agent Pipeline

**用户目标：** 用一个统一看板控制多设备、多类型项目中 agent 协同，支持任意数字生产 SOP 的维护和管理。

---

## 核心抽象（必须理解）

### 故事（Story）- 用户价值单元

**定义：** 从用户角度描述需求的工作单元，负责维护完整上下文并编排执行。

**关键特征：**
- 维护完整设计上下文（PRD、规范文档、资源引用）
- 拆解工作流为可执行任务（Tasks）
- 编排任务的执行顺序和依赖
- **不直接绑定 Agent 进程**
- 聚合任务结果并完成验收

**生命周期：**
```
创建 → 上下文注入 → 拆解完成 → 编排执行 → 验收完成/失败
```

**Story间关系：** 用户自定义的视图层概念，不影响执行流程。

---

### 任务（Task）- 执行容器

**定义：** 一对一绑定 Agent 进程的执行单元，负责在隔离环境中完成具体工作。

**关键特征：**
- 一对一绑定 Agent 进程（Claude Code、Codex、Gemini等）
- 在隔离环境（worktree/container/vm）中执行
- 从所属 Story 继承上下文 + 特定注入
- 捕获执行状态和产物，向 Story 报告

**生命周期：**
```
待分配 → 已分配 → 执行中 → 待验证 → 完成/失败
```

---

### Story vs Task 对比

| 维度 | 故事（Story） | 任务（Task） |
|------|------------|------------------|
| 定位 | 用户价值单元 | 执行容器 |
| 视角 | 用户视角（What） | 执行视角（How） |
| Agent绑定 | 不绑定 | 一对一绑定 |
| 上下文 | 维护完整设计信息 | 接收派生上下文 |
| 职责 | 描述、拆解、编排、验收 | 执行、捕获、报告 |
| 实例化 | 由用户或PM创建 | 由故事拆解时创建 |

## 系统架构

### 云端/本机双后端模型

AgentDashboard 采用**云端（Cloud）+ 本机（Local）**的双后端架构。

- **云端后端**：数据中枢 + 用户入口 + 调度中继 + 云端原生 Agent（PiAgent）。持有所有业务数据，暴露 REST API / MCP，通过 WebSocket 管理和调度本机。PiAgent 的 AgentLoop 在云端运行，直接访问业务数据，其 tool call 路由到本机执行。
- **本机后端**：per-machine 进程，管理本机上的多个 Workspace 目录。主动连接云端 WebSocket，接受命令：对第三方 Agent 执行完整的 `command.prompt`；对 PiAgent 执行 `command.tool.*`（文件读写、Shell 执行等）。注册时上报 `accessible_roots`（可访问目录列表）。

```
┌──────────────────────────────────────────────────────────┐
│                    云端后端（Cloud）                       │
│  ┌────────────────────────────────────────────────────┐  │
│  │  编排层 · 状态层 · 连接层                           │  │
│  │  REST API · MCP · SSE/NDJSON · WebSocket 服务端    │  │
    │  │  PiAgent AgentLoop（云端原生 Agent）               │  │
    │  │  BackendRegistry（在线本机管理）                     │  │
    │  │  Relay（命令路由 + 执行输出 + tool call 转发）      │  │
│  └────────────────────────────────────────────────────┘  │
└──────────────────────────┬───────────────────────────────┘
                           │ WebSocket（本机主动连接）
            ┌──────────────┼──────────────┐
            ▼              ▼              ▼
    ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
    │ 本机 A         │ │ 本机 B       │ │ 本机 C       │
    │ 第三方Agent执行│ │ 第三方Agent  │ │ 第三方Agent  │
    │ PiAgent工具执行│ │ PiAgent工具  │ │ PiAgent工具  │
    │ 工作空间文件   │ │ 工作空间文件 │ │ 工作空间文件 │
    └──────────────┘ └──────────────┘ └──────────────┘
```

> 三层逻辑（编排/状态/连接）全部运行在云端。PiAgent（云端原生 Agent）也运行在云端。  
> 本机是第三方 Agent 的执行器 + PiAgent 的远程工具环境。  
> 详见 `docs/relay-protocol.md` 和 `docs/modules/09-relay.md`

## 核心设计原则

### 1. 策略可插拔（最重要原则）

每个模块的实现细节都是可替换的：
- **状态存储**：文件 / 数据库 / 混合（由使用场景决定）
- **隔离方式**：worktree / 快照 / 容器 / VM（代码项目vs游戏项目不同需求）
- **编排策略**：规则驱动 / Agent PM / 手动 / 混合
- **验证方式**：脚本 / Agent审查 / 人工确认 / 组合

> **不预设最佳实践，提供可换的策略接口。**

### 2. 状态即真相与数据权属

- 所有操作都体现为状态变更。
- **所有业务数据归云端所有**：Project、Story、Task、Workspace 元数据（含 `backend_id` 标记物理文件所在本机）、Settings、StateChange。
- **本机只拥有执行状态**：Agent 进程、会话缓存、工作空间物理文件（一个本机可管理多个 Workspace 目录）。
- 状态快照可回溯、可对比。
- 历史轨迹完整记录（StateChange 不可变日志）。
- **云端**：同时承担数据管理、用户 API、MCP 暴露、看板视图、用户偏好等全部"中控"职责。本机通过 WebSocket 被云端调度。

### 3. 接口稳定，实现可变

- 模块间通过稳定的接口交互。
- 核心接口定义见各模块文档。
- 统一使用 **Agent Client Protocol / MCP** 定义的数据交换格式和结构，确保异构 Agent 产出的 Artifacts 语义一致。

### 4. 连接透明化与可恢复性（Resume）

- 用户无感知切换本地/远程后端。
- 支持多对多：一用户多后端、一后端多用户。
- **Resume 机制**：交互协议必须支持基于 ID 的增量恢复，确保网络断连或重启后能无缝衔接执行状态。

## 多层协作模型

```
用户 ──→ 浏览器 ──→ 云端后端 (Cloud) ──→ 本机后端群 (Locals) ──→ 工作空间
  ↕         ↕            ↕                      ↕
多用户    多设备    数据中枢/API/MCP         Agent执行工人
                   WebSocket 服务端          WebSocket 客户端
```

- 用户通过浏览器访问云端
- 云端持有全部业务数据，暴露 REST API 和 MCP
- 云端通过 WebSocket 管理多个本机（开发者的不同设备）
- 每个本机 = 一台机器，可管理多个 Workspace 目录
- 命令路由基于 `Workspace.backend_id`（物理文件所在本机），不依赖 Story/Project 上的 backend_id

## 技术栈

**已确定：**
- **后端语言/框架**：Rust + Axum + Tokio + SQLx
- **前端**：React 19 + TypeScript + Vite 7 + Tailwind v4 + Zustand
- **通信**：REST + SSE/NDJSON（前端↔云端）+ WebSocket（云端↔本机）
- **协议**：Agent Client Protocol (ACP) + MCP
- **数据库**：SQLite（预研阶段）
- **云端↔本机通信协议**：JSON over WebSocket（详见 `docs/relay-protocol.md`）
- **Agent 集成**：
  - **云端原生**：PiAgent（AgentLoop 在云端，tool call 路由到本机）
  - **本地第三方**：Claude Code、Codex、AMP 等（通过 AgentConnector trait 可扩展）

> 详见 `.trellis/spec/tech-stack.md`
