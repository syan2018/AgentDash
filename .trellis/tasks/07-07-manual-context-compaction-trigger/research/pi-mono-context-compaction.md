## Key Findings

1. pi-mono's primary coding-agent compaction model is append-only session entries plus a synthetic summary, not event-log projection replacement.

   Evidence: `CompactionEntry` stores `summary`, `firstKeptEntryId`, `tokensBefore`, optional `details`, and `fromHook` in the session entry stream (`references/pi-mono/packages/coding-agent/src/core/session-manager.ts:66`, `references/pi-mono/packages/coding-agent/src/core/session-manager.ts:69`, `references/pi-mono/packages/coding-agent/src/core/session-manager.ts:70`, `references/pi-mono/packages/coding-agent/src/core/session-manager.ts:72`, `references/pi-mono/packages/coding-agent/src/core/session-manager.ts:74`). `appendCompaction()` appends a new `compaction` entry as child of the current leaf and advances the leaf (`references/pi-mono/packages/coding-agent/src/core/session-manager.ts:868`, `references/pi-mono/packages/coding-agent/src/core/session-manager.ts:876`, `references/pi-mono/packages/coding-agent/src/core/session-manager.ts:887`).

2. The model-visible context is rebuilt as: latest compaction summary first, kept entries from `firstKeptEntryId`, then post-compaction entries.

   Evidence: `buildSessionContext()` comments describe exactly this order (`references/pi-mono/packages/coding-agent/src/core/session-manager.ts:366`, `references/pi-mono/packages/coding-agent/src/core/session-manager.ts:367`, `references/pi-mono/packages/coding-agent/src/core/session-manager.ts:368`, `references/pi-mono/packages/coding-agent/src/core/session-manager.ts:369`, `references/pi-mono/packages/coding-agent/src/core/session-manager.ts:370`). The implementation pushes `createCompactionSummaryMessage(...)` first (`references/pi-mono/packages/coding-agent/src/core/session-manager.ts:385`, `references/pi-mono/packages/coding-agent/src/core/session-manager.ts:387`), then emits kept messages once `entry.id === compaction.firstKeptEntryId` (`references/pi-mono/packages/coding-agent/src/core/session-manager.ts:392`, `references/pi-mono/packages/coding-agent/src/core/session-manager.ts:396`, `references/pi-mono/packages/coding-agent/src/core/session-manager.ts:399`), then emits later entries (`references/pi-mono/packages/coding-agent/src/core/session-manager.ts:404`, `references/pi-mono/packages/coding-agent/src/core/session-manager.ts:405`).

3. The synthetic summary is passed to the model as a user-role message with explicit "conversation history before this point" text.

   Evidence: `COMPACTION_SUMMARY_PREFIX` says "The conversation history before this point was compacted into the following summary:" (`references/pi-mono/packages/coding-agent/src/core/messages.ts:11`). `CompactionSummaryMessage` is a custom role with `summary` and `tokensBefore` (`references/pi-mono/packages/coding-agent/src/core/messages.ts:62`, `references/pi-mono/packages/coding-agent/src/core/messages.ts:64`, `references/pi-mono/packages/coding-agent/src/core/messages.ts:65`). `convertToLlm()` maps `compactionSummary` to a `role: "user"` message containing prefix + summary + suffix (`references/pi-mono/packages/coding-agent/src/core/messages.ts:176`, `references/pi-mono/packages/coding-agent/src/core/messages.ts:178`, `references/pi-mono/packages/coding-agent/src/core/messages.ts:180`).

4. Auto-compaction threshold is simple: compact when `contextTokens > contextWindow - reserveTokens`, with default `reserveTokens = 16384`, `keepRecentTokens = 20000`, and `enabled = true`.

   Evidence: defaults are declared in `DEFAULT_COMPACTION_SETTINGS` (`references/pi-mono/packages/coding-agent/src/core/compaction/compaction.ts:121`, `references/pi-mono/packages/coding-agent/src/core/compaction/compaction.ts:122`, `references/pi-mono/packages/coding-agent/src/core/compaction/compaction.ts:123`, `references/pi-mono/packages/coding-agent/src/core/compaction/compaction.ts:124`). `shouldCompact()` returns false when disabled and otherwise checks `contextTokens > contextWindow - settings.reserveTokens` (`references/pi-mono/packages/coding-agent/src/core/compaction/compaction.ts:219`, `references/pi-mono/packages/coding-agent/src/core/compaction/compaction.ts:220`, `references/pi-mono/packages/coding-agent/src/core/compaction/compaction.ts:221`). Settings expose the same defaults (`references/pi-mono/packages/coding-agent/src/core/settings-manager.ts:617`, `references/pi-mono/packages/coding-agent/src/core/settings-manager.ts:630`, `references/pi-mono/packages/coding-agent/src/core/settings-manager.ts:634`, `references/pi-mono/packages/coding-agent/src/core/settings-manager.ts:638`).

5. Auto-compaction has two built-in reasons: `overflow` and `threshold`; manual compaction reason is `manual`.

## Two Trigger Modes

Found in pi-mono:

- Automatic threshold mode: after `agent_end`, `_checkCompaction()` checks context pressure and calls `_runAutoCompaction("threshold", false)` when `shouldCompact()` is true (`references/pi-mono/packages/coding-agent/src/core/agent-session.ts:556`, `references/pi-mono/packages/coding-agent/src/core/agent-session.ts:568`, `references/pi-mono/packages/coding-agent/src/core/agent-session.ts:1814`, `references/pi-mono/packages/coding-agent/src/core/agent-session.ts:1815`).
- Automatic overflow recovery mode: if the assistant message is a same-model context overflow error, pi removes the error from agent context, runs `_runAutoCompaction("overflow", true)`, then retries via `Agent.continue()` (`references/pi-mono/packages/coding-agent/src/core/agent-session.ts:1765`, `references/pi-mono/packages/coding-agent/src/core/agent-session.ts:1766`, `references/pi-mono/packages/coding-agent/src/core/agent-session.ts:1781`, `references/pi-mono/packages/coding-agent/src/core/agent-session.ts:1785`, `references/pi-mono/packages/coding-agent/src/core/agent-session.ts:1787`, `references/pi-mono/packages/coding-agent/src/core/agent-session.ts:1960`, `references/pi-mono/packages/coding-agent/src/core/agent-session.ts:1968`).
- Pre-prompt auto check: before sending a prompt, pi checks the last assistant message for compaction, catching aborted responses near the limit (`references/pi-mono/packages/coding-agent/src/core/agent-session.ts:1009`, `references/pi-mono/packages/coding-agent/src/core/agent-session.ts:1010`, `references/pi-mono/packages/coding-agent/src/core/agent-session.ts:1012`). Changelog says this was added to catch user-aborted mid-response cases (`references/pi-mono/packages/coding-agent/CHANGELOG.md:2934`).
- Manual `/compact`: immediate call to `session.compact(customInstructions)`, with optional summary focus instructions (`references/pi-mono/packages/coding-agent/docs/compaction.md:37`, `references/pi-mono/packages/coding-agent/src/modes/interactive/interactive-mode.ts:2186`, `references/pi-mono/packages/coding-agent/src/modes/interactive/interactive-mode.ts:2187`, `references/pi-mono/packages/coding-agent/src/modes/interactive/interactive-mode.ts:4667`).
- Extension-triggered manual-like compaction: `ctx.compact()` does not await completion and supports callbacks (`references/pi-mono/packages/coding-agent/src/core/extensions/types.ts:256`, `references/pi-mono/packages/coding-agent/src/core/extensions/types.ts:258`, `references/pi-mono/packages/coding-agent/src/core/extensions/types.ts:290`, `references/pi-mono/packages/coding-agent/src/core/extensions/types.ts:291`). Example `trigger-compact` calls `ctx.compact()` on a threshold crossing and also registers `/trigger-compact` (`references/pi-mono/packages/coding-agent/examples/extensions/trigger-compact.ts:8`, `references/pi-mono/packages/coding-agent/examples/extensions/trigger-compact.ts:12`, `references/pi-mono/packages/coding-agent/examples/extensions/trigger-compact.ts:27`, `references/pi-mono/packages/coding-agent/examples/extensions/trigger-compact.ts:34`, `references/pi-mono/packages/coding-agent/examples/extensions/trigger-compact.ts:40`, `references/pi-mono/packages/coding-agent/examples/extensions/trigger-compact.ts:43`, `references/pi-mono/packages/coding-agent/examples/extensions/trigger-compact.ts:47`).

Not found in inspected scope:

- No AgentDashboard-like split between "running active: schedule next turn" and "idle: launch compact-only turn" was found.
- No durable pending manual compaction request table/queue was found.
- No compact-only turn source/mode was found.
- No command outcome names equivalent to `scheduled_next_turn`, `launched_compaction_turn`, `no_eligible_messages`, or `blocked` were found.

Important mismatch:

- pi-mono manual compaction aborts any current agent operation before compaction (`references/pi-mono/packages/coding-agent/src/core/agent-session.ts:1581`, `references/pi-mono/packages/coding-agent/src/core/agent-session.ts:1585`, `references/pi-mono/packages/coding-agent/src/core/agent-session.ts:1586`). AgentDashboard's task requires the running active turn not be interrupted, with the manual request consumed before the next provider call. pi-mono should not be copied for running-state behavior.

## Resume/Continuation Semantics

1. Resume loads model-visible messages from durable session entries via `buildSessionContext()`.

   Evidence: SDK startup checks for existing session data with `sessionManager.buildSessionContext()` (`references/pi-mono/packages/coding-agent/src/core/sdk.ts:189`, `references/pi-mono/packages/coding-agent/src/core/sdk.ts:190`, `references/pi-mono/packages/coding-agent/src/core/sdk.ts:191`) and restores `agent.state.messages = existingSession.messages` (`references/pi-mono/packages/coding-agent/src/core/sdk.ts:330`, `references/pi-mono/packages/coding-agent/src/core/sdk.ts:331`, `references/pi-mono/packages/coding-agent/src/core/sdk.ts:332`). Session switching opens the selected session and starts runtime with `session_start` reason `resume` (`references/pi-mono/packages/coding-agent/src/core/agent-session-runtime.ts:128`, `references/pi-mono/packages/coding-agent/src/core/agent-session-runtime.ts:135`, `references/pi-mono/packages/coding-agent/src/core/agent-session-runtime.ts:139`, `references/pi-mono/packages/coding-agent/src/core/agent-session-runtime.ts:143`).

2. The resume payload is the synthetic summary message plus retained suffix, not a system message or special continuation event.

   Evidence: `buildSessionContext()` pushes `compactionSummary` as a normal `AgentMessage` (`references/pi-mono/packages/coding-agent/src/core/session-manager.ts:385`, `references/pi-mono/packages/coding-agent/src/core/session-manager.ts:387`), and `convertToLlm()` converts it to user-role content (`references/pi-mono/packages/coding-agent/src/core/messages.ts:176`, `references/pi-mono/packages/coding-agent/src/core/messages.ts:178`, `references/pi-mono/packages/coding-agent/src/core/messages.ts:180`). No special resume-after-compact event was found in inspected scope.

3. After manual or auto compaction, the live agent state is rebuilt immediately from `buildSessionContext()`.

   Evidence: manual compact appends the compaction, builds session context, and assigns `this.agent.state.messages = sessionContext.messages` (`references/pi-mono/packages/coding-agent/src/core/agent-session.ts:1663`, `references/pi-mono/packages/coding-agent/src/core/agent-session.ts:1665`, `references/pi-mono/packages/coding-agent/src/core/agent-session.ts:1666`). Auto compact does the same (`references/pi-mono/packages/coding-agent/src/core/agent-session.ts:1934`, `references/pi-mono/packages/coding-agent/src/core/agent-session.ts:1936`, `references/pi-mono/packages/coding-agent/src/core/agent-session.ts:1937`).

4. Continuation after overflow or queued messages uses `Agent.continue()`, which resumes from existing context without adding a new user message.

   Evidence: base README says `continue()` resumes from existing context without adding a new message and is for retries after errors (`references/pi-mono/packages/agent/README.md:111`, `references/pi-mono/packages/agent/README.md:113`). `Agent.continue()` runs queued steering/follow-ups if the last message is assistant, otherwise calls `runContinuation()` (`references/pi-mono/packages/agent/src/agent.ts:333`, `references/pi-mono/packages/agent/src/agent.ts:334`, `references/pi-mono/packages/agent/src/agent.ts:340`, `references/pi-mono/packages/agent/src/agent.ts:349`, `references/pi-mono/packages/agent/src/agent.ts:387`, `references/pi-mono/packages/agent/src/agent.ts:389`). Auto compaction schedules `this.agent.continue()` after overflow retry or queued messages (`references/pi-mono/packages/coding-agent/src/core/agent-session.ts:1960`, `references/pi-mono/packages/coding-agent/src/core/agent-session.ts:1967`, `references/pi-mono/packages/coding-agent/src/core/agent-session.ts:1968`, `references/pi-mono/packages/coding-agent/src/core/agent-session.ts:1970`, `references/pi-mono/packages/coding-agent/src/core/agent-session.ts:1974`).

5. Interactive input during compaction is not sent immediately; it is queued and flushed after `compaction_end`.

   Evidence: interactive mode queues non-extension input while `session.isCompacting` (`references/pi-mono/packages/coding-agent/src/modes/interactive/interactive-mode.ts:2236`, `references/pi-mono/packages/coding-agent/src/modes/interactive/interactive-mode.ts:2237`, `references/pi-mono/packages/coding-agent/src/modes/interactive/interactive-mode.ts:2242`, `references/pi-mono/packages/coding-agent/src/modes/interactive/interactive-mode.ts:2243`). `queueCompactionMessage()` records text and status (`references/pi-mono/packages/coding-agent/src/modes/interactive/interactive-mode.ts:3187`, `references/pi-mono/packages/coding-agent/src/modes/interactive/interactive-mode.ts:3188`, `references/pi-mono/packages/coding-agent/src/modes/interactive/interactive-mode.ts:3192`). `compaction_end` flushes that queue (`references/pi-mono/packages/coding-agent/src/modes/interactive/interactive-mode.ts:2534`), and `flushCompactionQueue()` delivers queued messages as prompt/follow-up/steer depending on retry and mode (`references/pi-mono/packages/coding-agent/src/modes/interactive/interactive-mode.ts:3206`, `references/pi-mono/packages/coding-agent/src/modes/interactive/interactive-mode.ts:3227`, `references/pi-mono/packages/coding-agent/src/modes/interactive/interactive-mode.ts:3231`, `references/pi-mono/packages/coding-agent/src/modes/interactive/interactive-mode.ts:3233`, `references/pi-mono/packages/coding-agent/src/modes/interactive/interactive-mode.ts:3235`).

## Lessons For AgentDashboard

Can directly adopt or adapt:

- Keep the explicit `firstKept` boundary idea. pi-mono's `firstKeptEntryId` is the durable resume cursor for retained suffix reconstruction (`references/pi-mono/packages/coding-agent/src/core/session-manager.ts:69`, `references/pi-mono/packages/coding-agent/src/core/session-manager.ts:392`, `references/pi-mono/packages/coding-agent/src/core/session-manager.ts:396`). AgentDashboard already has `first_kept_ref`; this reinforces making it mandatory for successful structural compaction.
- Keep "force bypasses threshold, not cut validity." pi-mono manual `/compact` bypasses token threshold but still runs `prepareCompaction()` and errors when no preparation exists (`references/pi-mono/packages/coding-agent/src/core/agent-session.ts:1600`, `references/pi-mono/packages/coding-agent/src/core/agent-session.ts:1601`, `references/pi-mono/packages/coding-agent/src/core/agent-session.ts:1607`). AgentDashboard should return no-op rather than writing an incomplete projection if no legal boundary exists.
- Use summary-as-data plus retained suffix as the recovery invariant. pi-mono makes summary a structured persisted entry and rebuilds model context from it (`references/pi-mono/packages/coding-agent/src/core/session-manager.ts:66`, `references/pi-mono/packages/coding-agent/src/core/session-manager.ts:385`, `references/pi-mono/packages/coding-agent/src/core/session-manager.ts:387`). AgentDashboard should keep the stronger projection-head version of this invariant.
- Recalculate UI context usage after compaction instead of trusting pre-compaction usage. pi-mono treats usage as unknown until a post-compaction response (`references/pi-mono/packages/coding-agent/src/core/agent-session.ts:2912`, `references/pi-mono/packages/coding-agent/src/core/agent-session.ts:2936`, `references/pi-mono/packages/coding-agent/src/core/agent-session.ts:2937`). AgentDashboard can instead show the projection's estimated tokens, but should distinguish estimates from provider usage.
- Add extension/hook-like before/after compaction diagnostics only if they preserve core projection ownership. pi-mono's `session_before_compact` and `session_compact` are useful observability/customization seams (`references/pi-mono/packages/coding-agent/src/core/extensions/types.ts:466`, `references/pi-mono/packages/coding-agent/src/core/extensions/types.ts:475`, `references/pi-mono/packages/coding-agent/src/core/extensions/types.ts:476`).
- Consider file-operation/provenance tags in generated summary. pi-mono accumulates read/modified files into summary tags (`references/pi-mono/packages/coding-agent/src/core/compaction/utils.ts:72`, `references/pi-mono/packages/coding-agent/src/core/compaction/utils.ts:75`, `references/pi-mono/packages/coding-agent/src/core/compaction/utils.ts:78`; docs at `references/pi-mono/packages/coding-agent/docs/compaction.md:178`, `references/pi-mono/packages/coding-agent/docs/compaction.md:180`, `references/pi-mono/packages/coding-agent/docs/compaction.md:184`). AgentDashboard's lifecycle recall surface can support this more rigorously through context frames / VFS summaries.

Not suitable to copy:

- Do not copy pi-mono's manual compaction behavior for running sessions. It aborts the active operation (`references/pi-mono/packages/coding-agent/src/core/agent-session.ts:1581`, `references/pi-mono/packages/coding-agent/src/core/agent-session.ts:1585`, `references/pi-mono/packages/coding-agent/src/core/agent-session.ts:1586`), while AgentDashboard requires "current turn continues; next pre-provider boundary consumes request."
- Do not copy append-only session-entry compaction as-is. AgentDashboard needs `runtime_session_compactions`, projection segments, projection head, and `context_compacted` eventing; pi-mono's model has no projection head or `compacted_until_ref`.
- Do not use a user-role synthetic summary as the only boundary if AgentDashboard can preserve explicit `origin=projection`, `active_compaction_id`, and source ranges. pi-mono's summary is user-role content (`references/pi-mono/packages/coding-agent/src/core/messages.ts:176`, `references/pi-mono/packages/coding-agent/src/core/messages.ts:178`); AgentDashboard's spec wants provenance-rich projection messages.
- Do not treat `Agent.continue()` as equivalent to compact-only. pi-mono `continue()` resumes normal agent operation from existing context (`references/pi-mono/packages/agent/README.md:111`, `references/pi-mono/packages/agent/README.md:113`); it is not a turn whose only purpose is compaction.
- Do not omit idempotent command receipt semantics. pi-mono's `/compact` and extension `ctx.compact()` did not show durable `client_command_id` replay protection in inspected scope.

## Suggested Improvements Beyond Manual Trigger

- Add a post-compaction context-usage state that distinguishes provider-verified usage, local projection estimate, and unknown. pi-mono only exposes `null` until the next response (`references/pi-mono/packages/coding-agent/src/core/agent-session.ts:2936`, `references/pi-mono/packages/coding-agent/src/core/agent-session.ts:2937`); AgentDashboard can do better because projection view already estimates model context composition.
- Carry summary-generation diagnostics into `context_compacted` / `context_compaction_failed`: summary model, cut boundary, messages summarized, messages kept, and no-op reason. pi-mono emits reason/result/error on `compaction_end` (`references/pi-mono/packages/coding-agent/src/core/agent-session.ts:121`, `references/pi-mono/packages/coding-agent/src/core/agent-session.ts:122`, `references/pi-mono/packages/coding-agent/src/core/agent-session.ts:123`, `references/pi-mono/packages/coding-agent/src/core/agent-session.ts:126`); AgentDashboard can make those durable projection diagnostics.
- Preserve split-turn prefix semantics. pi-mono has explicit split-turn summary handling (`references/pi-mono/packages/coding-agent/src/core/compaction/compaction.ts:658`, `references/pi-mono/packages/coding-agent/src/core/compaction/compaction.ts:660`, `references/pi-mono/packages/coding-agent/src/core/compaction/compaction.ts:752`, `references/pi-mono/packages/coding-agent/src/core/compaction/compaction.ts:755`). AgentDashboard's cut-point algorithm should make mid-turn behavior explicit in metadata if it ever supports such cuts.
- Add summary input bounding as a first-class rule. pi-mono truncates tool results to 2000 chars before summarization (`references/pi-mono/packages/coding-agent/src/core/compaction/utils.ts:88`, `references/pi-mono/packages/coding-agent/src/core/compaction/utils.ts:156`) and tests it (`references/pi-mono/packages/coding-agent/test/compaction-serialization.test.ts:21`, `references/pi-mono/packages/coding-agent/test/compaction-serialization.test.ts:22`). AgentDashboard already has bounded tool facts; summarizer should consume those bounded facts rather than rehydrating large bodies.
- Add tests for stale pre-compaction usage not retriggering compaction. pi-mono has explicit coverage that a pre-compaction assistant usage should not cause another compaction (`references/pi-mono/packages/coding-agent/test/agent-session-auto-compaction-queue.test.ts:181`, `references/pi-mono/packages/coding-agent/test/agent-session-auto-compaction-queue.test.ts:209`, `references/pi-mono/packages/coding-agent/test/agent-session-auto-compaction-queue.test.ts:233`, `references/pi-mono/packages/coding-agent/test/agent-session-auto-compaction-queue.test.ts:235`). AgentDashboard should cover the equivalent projection-head/usage refresh path.
- Add a "history source of truth vs model context projection" explanatory doc. Mom documents `log.jsonl` as never compacted and `context.jsonl` as compacted LLM context (`references/pi-mono/packages/mom/README.md:371`, `references/pi-mono/packages/mom/README.md:376`, `references/pi-mono/packages/mom/README.md:379`, `references/pi-mono/packages/mom/README.md:382`). AgentDashboard's equivalent is `session_events` vs `session_projection_heads/segments`; this would help future agents avoid confusing timeline history with model input.
