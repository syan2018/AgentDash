### Code Patterns

- `HookRuntimeAccess` 仍同时暴露 `session_id()` 与 `control_target()`；`session_id` 是 runtime adapter/delivery provenance，`control_target` 是业务 owner：`crates/agentdash-spi/src/hooks/mod.rs:482`。
- `HookControlTarget` 是 `run_id + agent_id + frame_id` 三元组，SPI 已具备 frame-first owner 类型：`crates/agentdash-spi/src/hooks/mod.rs:560`。
- `ExecutionHookProvider` 的 frame-first 方法与 `resolve_runtime_hook_target(runtime_session_id)` adapter 方法并存：`crates/agentdash-spi/src/hooks/mod.rs:953`。
- `AgentFrameHookRuntime` 文件头明确“以 agent/frame 为主键”，`runtime_session_id` 仅作 trace adapter/provider query：`crates/agentdash-application/src/workflow/frame_hook_runtime.rs:36`。
- `AgentFrameHookRuntime::from_frame` 从 `run_id` + `AgentFrame` + delivery session 创建 runtime scope：`crates/agentdash-application/src/workflow/frame_hook_runtime.rs:133`。
- `AgentFrameHookRuntime::control_target` 返回 `HookControlTarget`，refresh/evaluate 走 frame provider entry：`crates/agentdash-application/src/workflow/frame_hook_runtime.rs:216`、`crates/agentdash-application/src/workflow/frame_hook_runtime.rs:263`、`crates/agentdash-application/src/workflow/frame_hook_runtime.rs:307`。
- `SessionRuntimeRegistry` 仍以 `HashMap<String, SessionRuntime>` 存储 runtime entry，`hook_runtime(session_id)` 和 `set_or_replace_hook_runtime(session_id, ...)` 是 delivery-session cache：`crates/agentdash-application/src/session/runtime_registry.rs:11`、`crates/agentdash-application/src/session/runtime_registry.rs:93`、`crates/agentdash-application/src/session/runtime_registry.rs:100`。
- `SessionRuntime` 仍有 `hook_runtime: Option<SharedHookRuntime>`，注释称 session 级跨 turn 共享：`crates/agentdash-application/src/session/hub_support.rs:207`。
- `SessionHookService::ensure_hook_runtime_for_target` 已要求 `AgentFrameRuntimeTarget`，但内部仍先通过 delivery session lazy ensure，再校验 target：`crates/agentdash-application/src/session/hooks_service.rs:25`、`crates/agentdash-application/src/session/hooks_service.rs:37`。
- `validate_hook_runtime_target` 明确检查 runtime session 与 frame_id 两者一致，target mismatch 目前在调用方暴露为错误：`crates/agentdash-application/src/session/hooks_service.rs:196`。
- `build_frame_hook_runtime` 校验 target frame agent 与 delivery anchor agent 一致，再 `AgentFrameHookRuntime::from_frame`：`crates/agentdash-application/src/session/hooks_service.rs:234`。
- `SessionRuntimeInner::ensure_hook_runtime_for_delivery_session` 注释为 delivery adapter，业务路径应走 `SessionHookService::ensure_hook_runtime_for_target`：`crates/agentdash-application/src/session/hub/hook_dispatch.rs:188`。
- `SessionRuntimeInner::get_hook_runtime_by_delivery_session` 也注释为 delivery adapter，仅供 hub 内部 adapter 场景：`crates/agentdash-application/src/session/hub/facade.rs:53`。
- `AgentFrameRuntimeTarget` 的 `frame_id` 表达 effective runtime surface，`delivery_runtime_session_id` 用于 live connector/runtime registry 同步：`crates/agentdash-application/src/session/types.rs:63`。
- `resolve_runtime_session_frame_id` 注释要求后续 command 携带 `AgentFrameRuntimeTarget`，不再把 raw session id 当 frame revision 写入目标：`crates/agentdash-application/src/session/hub/tool_builder.rs:124`。
- `replace_current_capability_state` 写入路径是 `AgentFrame revision -> memory cache -> connector sync`：`crates/agentdash-application/src/session/hub/tool_builder.rs:168`。
- `apply_live_runtime_context_transition` 使用 `target_frame_id + delivery_runtime_session_id` 组成 `AgentFrameRuntimeTarget`，再更新 hook capabilities 和 context frame：`crates/agentdash-application/src/session/hub/runtime_context_transition.rs:101`。
- Canvas exposure 会写 `visible_canvas_mount`、同步 VFS capability state、再写 `visible_workspace_module_ref`：`crates/agentdash-application/src/canvas/tools.rs:242`、`crates/agentdash-application/src/canvas/tools.rs:289`。
- `workspace_module_present` 对 canvas renderer 先 `expose_existing_canvas_for_session`，然后才注入 `workspace_module_presented`：`crates/agentdash-application/src/workspace_module/tools.rs:1006`、`crates/agentdash-application/src/workspace_module/tools.rs:1034`。
- launch commit 在 accepted 后写 AgentFrame revision，并调用 `sync_hook_runtime_target` 确保 runtime target 到新 frame：`crates/agentdash-application/src/session/launch/commit.rs:144`、`crates/agentdash-application/src/session/launch/commit.rs:255`。
- `resolve_current_frame_for_runtime_session` 通过 anchor -> LifecycleAgent -> current frame，找不到 current frame 才用 launch frame：`crates/agentdash-application/src/workflow/session_association.rs:34`。
- companion tool context 从 `context.turn.hook_runtime.session_id()` 取 delivery session，再解析 lifecycle anchor：`crates/agentdash-application/src/companion/tool_context.rs:26`、`crates/agentdash-application/src/companion/tool_context.rs:97`。
- companion gate control 使用 gate owner frame/agent 与 anchor 选择 delivery runtime session；Specific policy 会校验 session anchor agent：`crates/agentdash-application/src/companion/gate_control.rs:660`、`crates/agentdash-application/src/companion/gate_control.rs:698`。
- `complete_lifecycle_node` 仍把 `snapshot.runtime_adapter_session_id` 传给 orchestrator：`crates/agentdash-application/src/workflow/tools/advance_node.rs:119`。

### High-Risk Migration Notes

- `SessionRuntimeRegistry` 和 `SessionRuntime.hook_runtime` 是最容易被误读为 RuntimeSession-first owner 的剩余结构。当前依赖 `SessionHookService` 和 lazy rebuild 把 stale runtime 替换掉，短期可用，但类型命名仍没有表达“delivery binding”。
- `SessionHookService::ensure_hook_runtime_for_target` 已是正确对外入口，但内部流程仍可能把 target mismatch 暴露给上层工具。迁移目标应让 service 层吸收 mismatch 并完成 rebind / invalidate / rebuild。
- Canvas / workspace module 已基本沿 AgentFrame capability transition 写事实，但 API 形态仍传 `session_id`。建议在工具构造或 service adapter 边界预解析 `AgentFrameRuntimeTarget`，工具主体只消费 target。
- launch commit 是已存在的关键同步点：accepted AgentFrame revision 后调用 `ensure_hook_runtime_for_target`。这个 warn-only 同步失败策略是最终 cutover gate 需要重点决策的地方。
- companion/gate 回流总体比 Canvas 更接近目标模型：LifecycleGate 保存 agent/frame owner，delivery runtime session 只是通知通道。主要迁移点是 `CompanionToolContext` 从 `hook_runtime.session_id()` 起步，应优先使用 `hook_runtime.control_target()`。
