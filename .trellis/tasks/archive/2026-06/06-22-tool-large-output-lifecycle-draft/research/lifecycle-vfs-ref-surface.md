### 2. `fs_read` defense mechanism details

- Schema denies unknown fields and requires only `path`; `offset` and `limit` are optional (`crates/agentdash-application/src/vfs/tools/fs/read.rs:91`, `crates/agentdash-application/src/vfs/tools/fs/read.rs:93`).
- URI/default-mount resolution rejects empty path and ambiguous unqualified path; VFS parser normalizes path before returning (`crates/agentdash-application/src/vfs/tools/common.rs:15`, `crates/agentdash-application/src/vfs/tools/common.rs:17`, `crates/agentdash-application/src/vfs/tools/common.rs:35`, `crates/agentdash-application/src/vfs/path.rs:193`, `crates/agentdash-application/src/vfs/path.rs:228`).
- Mount IDs reject URI separators, slash/backslash and whitespace (`crates/agentdash-application/src/vfs/path.rs:16`, `crates/agentdash-application/src/vfs/path.rs:21`).
- Mount-relative paths reject empty when not allowed, absolute-like paths including Windows drive forms, and `..` escape beyond mount root (`crates/agentdash-application/src/vfs/path.rs:298`, `crates/agentdash-application/src/vfs/path.rs:308`, `crates/agentdash-application/src/vfs/path.rs:317`, `crates/agentdash-application/src/vfs/path.rs:337`).
- VFS links resolve only up to `MAX_LINK_DEPTH = 5`, then fail to prevent cycles/long chains (`crates/agentdash-application/src/vfs/path.rs:9`, `crates/agentdash-application/src/vfs/path.rs:199`, `crates/agentdash-application/src/vfs/path.rs:232`).
- Provider dispatch verifies mount existence and requested capability before resolving provider and calling SPI (`crates/agentdash-application/src/vfs/path.rs:246`, `crates/agentdash-application/src/vfs/path.rs:251`, `crates/agentdash-application/src/vfs/path.rs:256`, `crates/agentdash-application/src/vfs/service.rs:76`, `crates/agentdash-application/src/vfs/service.rs:85`).
- Before text read, `fs_read` calls `stat`; if runtime entry is binary, it routes to `read_binary_entry` instead of attempting text (`crates/agentdash-application/src/vfs/tools/fs/read.rs:136`, `crates/agentdash-application/src/vfs/tools/fs/read.rs:137`, `crates/agentdash-application/src/vfs/tools/fs/read.rs:146`, `crates/agentdash-application/src/vfs/tools/fs/read.rs:148`).
- Non-image binary returns `is_error = true` with `type = "unsupported_binary"`; image binary returns a metadata text block plus `ContentPart::Image` with base64 data (`crates/agentdash-application/src/vfs/tools/fs/read.rs:313`, `crates/agentdash-application/src/vfs/tools/fs/read.rs:314`, `crates/agentdash-application/src/vfs/tools/fs/read.rs:320`, `crates/agentdash-application/src/vfs/tools/fs/read.rs:328`, `crates/agentdash-application/src/vfs/tools/fs/read.rs:338`, `crates/agentdash-application/src/vfs/tools/fs/read.rs:340`).
- Tool offset is 1-based; SPI offset is 0-based via `saturating_sub(1)`, so `offset = 0` behaves like start-of-file (`crates/agentdash-application/src/vfs/tools/fs/read.rs:151`, `crates/agentdash-application/src/vfs/tools/fs/read.rs:152`).
- ENOENT is special-cased: `MountError::NotFound` triggers up to three `suggest_paths` candidates and returns a friendly `ExecutionFailed` message (`crates/agentdash-application/src/vfs/tools/fs/read.rs:166`, `crates/agentdash-application/src/vfs/tools/fs/read.rs:167`, `crates/agentdash-application/src/vfs/tools/fs/read.rs:175`, `crates/agentdash-application/src/vfs/tools/fs/read.rs:177`).
- Large-file guard only runs when `limit` is omitted. It rejects full text results above 256 KiB or 5000 lines with `AgentToolResult { is_error: true }` and structured details (`crates/agentdash-application/src/vfs/tools/fs/read.rs:26`, `crates/agentdash-application/src/vfs/tools/fs/read.rs:27`, `crates/agentdash-application/src/vfs/tools/fs/read.rs:28`, `crates/agentdash-application/src/vfs/tools/fs/read.rs:186`, `crates/agentdash-application/src/vfs/tools/fs/read.rs:188`, `crates/agentdash-application/src/vfs/tools/fs/read.rs:191`, `crates/agentdash-application/src/vfs/tools/fs/read.rs:234`, `crates/agentdash-application/src/vfs/tools/fs/read.rs:250`).
- Dedup is per `FsReadTool` instance using `LruCache` size 64 keyed by `(mount_id, path, offset, limit)` and provider `version_token`; repeat reads with identical token return a short `[unchanged since previous read]` stub (`crates/agentdash-application/src/vfs/tools/fs/read.rs:30`, `crates/agentdash-application/src/vfs/tools/fs/read.rs:34`, `crates/agentdash-application/src/vfs/tools/fs/read.rs:47`, `crates/agentdash-application/src/vfs/tools/fs/read.rs:197`, `crates/agentdash-application/src/vfs/tools/fs/read.rs:204`, `crates/agentdash-application/src/vfs/tools/fs/read.rs:208`, `crates/agentdash-application/src/vfs/tools/fs/read.rs:214`, `crates/agentdash-application/src/vfs/tools/fs/read.rs:266`).
- Output text is `cat -n` style with absolute 1-based line numbers and a `file: {path}` header (`crates/agentdash-application/src/vfs/tools/fs/read.rs:218`, `crates/agentdash-application/src/vfs/tools/fs/read.rs:219`, `crates/agentdash-application/src/vfs/tools/fs/read.rs:226`, `crates/agentdash-application/src/vfs/tools/fs/read.rs:230`).

### 3. Most natural paths for tool-result refs

The natural location is under the existing session evidence tree, not under writable `artifacts`:

- Current tool calls are already session items exposed under `session/tools/{item-file}.json` and filtered by `SessionItemView::Tools` (`crates/agentdash-application/src/vfs/provider_lifecycle.rs:155`, `crates/agentdash-application/src/lifecycle/surface/journey/session_items.rs:281`, `crates/agentdash-application/src/lifecycle/surface/journey/session_items.rs:310`).
- Native `fs_read` item targets already preserve the read path for deterministic tool file naming (`crates/agentdash-application/src/lifecycle/surface/journey/session_items.rs:561`).
- `session/writes` is intentionally scoped to successful write tools (`crates/agentdash-application/src/lifecycle/surface/journey/session_items.rs:282`, `crates/agentdash-application/src/lifecycle/surface/journey/session_items.rs:584`), so large read/result refs should not be hung there unless they are write-result evidence.
- `session/terminal` already aggregates shell output separately from item JSON (`crates/agentdash-application/src/lifecycle/surface/journey/mod.rs:168`, `crates/agentdash-application/src/lifecycle/surface/journey/mod.rs:172`), so shell large-output refs naturally belong below a terminal/tool-output sub-tree rather than inside `artifacts`.
- `node/artifacts` / `artifacts` are current node output ports and enforce `writable_port_keys`; using them for arbitrary tool result refs would confuse workflow contract outputs with trace evidence (`crates/agentdash-application/src/vfs/provider_lifecycle.rs:641`, `crates/agentdash-application/src/vfs/provider_lifecycle.rs:654`, `.trellis/spec/backend/workflow/activity-lifecycle.md:174`, `.trellis/spec/backend/vfs/vfs-access.md:262`).
- `records` are user/agent journey notes scoped by encoded node path (`crates/agentdash-application/src/vfs/provider_lifecycle.rs:483`, `crates/agentdash-application/src/lifecycle/surface/journey/mod.rs:327`, `crates/agentdash-application/src/lifecycle/surface/journey/mod.rs:383`), not raw tool-result blob storage.

Recommended shape:

- Add a read-only subtree under `session/tools`, e.g. `session/tools/{item-file}.json` keeps metadata and references, while `session/tools/{item-id}/result.{txt,json}` or `session/tool-results/{item-id}/...` stores/read-projects large result bodies. The latter is cleaner because current `session/tools` is a flat list of files, and `list_session_item_entries` currently emits only files, not per-tool directories (`crates/agentdash-application/src/vfs/provider_lifecycle.rs:874`, `crates/agentdash-application/src/vfs/provider_lifecycle.rs:884`).
- For shell output, either continue exposing aggregated `session/terminal` and add segmented refs under `session/terminal/{item-id}.txt`, or put the per-tool body under the same `session/tool-results/{item-id}/stdout.txt` surface and keep `session/terminal` as aggregate convenience.
- For node-scoped runtime views, mirror the same session-relative paths under `lifecycle://session/...`; `node_runtime` already resolves `session/*` from the current node's `ExecutorRunRef::RuntimeSession` (`crates/agentdash-application/src/vfs/provider_lifecycle.rs:572`, `crates/agentdash-application/src/vfs/provider_lifecycle.rs:574`).
- If a large tool result becomes declared workflow output, it should be explicitly copied/materialized to `artifacts/{port_key}` by the runtime/tool logic, not implicitly mounted there. That preserves `writable_port_keys` and output-port semantics (`crates/agentdash-application/src/vfs/provider_lifecycle.rs:641`, `crates/agentdash-application/src/vfs/provider_lifecycle.rs:660`).

### 4. Where to express cache miss / expired

Layering recommendation:

- VFS/provider layer should express missing or expired ref content as a read result error on the ref path, using `MountError::NotFound` for unavailable path and `OperationFailed` only for invalid provider state. Existing journey errors already map `LifecycleJourneyError::NotFound` to `MountError::NotFound` (`crates/agentdash-application/src/vfs/provider_lifecycle.rs:253`, `crates/agentdash-application/src/vfs/provider_lifecycle.rs:255`).
- `fs_read` should convert `NotFound` into the existing friendly `File not found` path with suggestions; if the path is a ref path, the provider text should make the missing reason visible through path/message, but the tool contract can remain unchanged (`crates/agentdash-application/src/vfs/tools/fs/read.rs:166`, `crates/agentdash-application/src/vfs/tools/fs/read.rs:177`).
- Session item metadata should be the durable place to represent ref lifecycle state (`available`, `expired`, `cache_miss`, `size_bytes`, `mime_type`, `ref_path`) because item JSON is already the index surface for tool evidence (`crates/agentdash-application/src/lifecycle/surface/journey/session_items.rs:327`, `crates/agentdash-application/src/lifecycle/surface/journey/session_items.rs:337`, `crates/agentdash-application/src/lifecycle/surface/journey/session_items.rs:345`).
- `fs_read` large-file guard should remain about VFS text size, not ref-cache lifecycle. It can return an instruction/ref when a provider offers one, but cache miss/expired belongs in the lifecycle/session projection provider that owns the ref path.
- `file_unchanged` dedup is tool-instance read optimization, not cache validity. Do not reuse its `details.type = "file_unchanged"` for result-ref cache state (`crates/agentdash-application/src/vfs/tools/fs/read.rs:197`, `crates/agentdash-application/src/vfs/tools/fs/read.rs:283`).

### 5. High-risk test points

- Runtime tool visibility: capability states should still gate `mounts_list/fs_read/fs_glob/fs_grep/fs_apply_patch/shell_exec`, and missing `context.session.vfs` should fail VFS provider construction without fabricating tools (`crates/agentdash-application/src/vfs/tools/factory.rs:50`, `crates/agentdash-application/src/runtime_tools/provider.rs:92`).
- URI/path hardening: `lifecycle://../x`, absolute paths, Windows drive paths, ambiguous unqualified paths, and VFS link cycles should stay rejected (`crates/agentdash-application/src/vfs/path.rs:308`, `crates/agentdash-application/src/vfs/path.rs:317`, `crates/agentdash-application/src/vfs/path.rs:337`, `crates/agentdash-application/src/vfs/tools/common.rs:35`, `crates/agentdash-application/src/vfs/path.rs:232`).
- `fs_read` full-read guard: no `limit` must error over 256 KiB / 5000 lines; any implementation that returns ref stubs must avoid bypassing current `is_error` expectations unless tests and contract are intentionally changed (`crates/agentdash-application/src/vfs/tools/fs/read.rs:186`, `crates/agentdash-application/src/vfs/tools/fs/read.rs:234`, `crates/agentdash-application/src/vfs/tools/fs/read.rs:250`).
- `fs_read` dedup: ref changes should alter `version_token` or avoid returning stale `[unchanged]` stubs for changed ref bodies (`crates/agentdash-application/src/vfs/tools/fs/read.rs:197`, `crates/agentdash-application/src/vfs/tools/fs/read.rs:204`).
- Lifecycle `agent_run_session` vs `node_runtime`: AgentRun workspace read-only evidence must reject direct writes, while node runtime must allow only `artifacts/{allowed_port}` and `records/*` (`crates/agentdash-application/src/vfs/provider_lifecycle.rs:630`, `crates/agentdash-application/src/vfs/provider_lifecycle.rs:641`, `crates/agentdash-application/src/vfs/provider_lifecycle.rs:654`, `crates/agentdash-application/src/vfs/provider_lifecycle.rs:677`).
- Session item file naming: adding directories under `session/tools` may collide with current flat-file listing assumptions; tests should cover both list and direct read paths (`crates/agentdash-application/src/vfs/provider_lifecycle.rs:874`, `crates/agentdash-application/src/lifecycle/surface/journey/session_items.rs:300`).
- Terminal output: current `session/terminal` returns `NotFound` when no command output exists; segmented terminal refs should preserve aggregate behavior and avoid changing empty-output semantics (`crates/agentdash-application/src/lifecycle/surface/journey/mod.rs:168`, `crates/agentdash-application/src/lifecycle/surface/journey/mod.rs:178`).
- Search behavior: lifecycle `search_text` recursively reads every listed non-dir file and skips read errors; large result-ref files could make grep expensive unless search either indexes metadata only or respects truncation/size limits (`crates/agentdash-application/src/vfs/provider_lifecycle.rs:823`, `crates/agentdash-application/src/vfs/provider_lifecycle.rs:841`, `crates/agentdash-application/src/vfs/provider_lifecycle.rs:842`).
- Workflow artifact semantics: large tool-result refs should not silently appear as output ports; tests should assert only declared `writable_port_keys` can be written/read as `artifacts/{port_key}` (`crates/agentdash-application/src/vfs/provider_lifecycle.rs:643`, `crates/agentdash-application/src/vfs/provider_lifecycle.rs:654`).

## Caveats / Not Found

- I did not find an existing `tool-result refs` storage model in lifecycle VFS; current surfaces expose raw session item JSON/markdown, terminal aggregate text, compaction summary markdown, node artifacts, and records.
- I did not inspect database migrations or frontend resource browser code because this research topic asked for runtime tool / lifecycle VFS read surface and no implementation change.
- `session/summary` and `session/conclusions` are listed by `session_root_entries`, but `LifecycleJourneyProjection::read_session_projection` currently has no explicit match arms for them in the inspected code; direct reads appear to fall through to `NotFound`.
- The spec path names mention older `vfs/tools/provider.rs` and `vfs/tools/vfs_provider.rs`; current code places those under `runtime_tools/provider.rs` and `runtime_tools/vfs_provider.rs`.
- No external references were needed; this artifact is based on repository code and Trellis specs only.
