# Research: API / RuntimeGateway consumers

- Query: review API and RuntimeGateway consumers of RuntimeSession / current runtime surface / RuntimeGateway actions, with emphasis on `RuntimeSessionMcpAccess`, `mcp.list_tools`, `mcp.call_tool`, current frame resolver, VFS/backend anchor resolution and `RuntimeSessionExecutionAnchor` usage.
- Scope: internal
- Date: 2026-06-24

## Findings

### Boundary Baseline

- API 层目标职责是鉴权、DTO 和错误映射，业务编排进入 application 层；见 `.trellis/spec/backend/architecture.md:13`。
- Session runtime live state 不回答 AgentRun 当前 runtime surface；Canvas / Extension / VFS / Terminal / RuntimeGateway MCP access 必须消费 AgentRun / Lifecycle current runtime surface query，而不是从 active turn cache、API helper 或 hub idle branch 拼装；见 `.trellis/spec/backend/session/runtime-execution-state.md:31`、`.trellis/spec/backend/session/runtime-execution-state.md:99`。
- RuntimeGateway MCP surface 来源应为 `runtime_session_id -> RuntimeSessionExecutionAnchor -> AgentRun runtime address -> current surface revision`；provider 不解析 current `AgentFrame`，不读 session hub idle fallback；见 `.trellis/spec/backend/runtime-gateway.md:56`、`.trellis/spec/backend/runtime-gateway.md:68`、`.trellis/spec/backend/runtime-gateway.md:75`。
- Canvas iframe 通过 Gateway 调用 Session Action 时，API route 必须重新校验 Session 与 Canvas Project 绑定；见 `.trellis/spec/backend/runtime-gateway.md:135`、`.trellis/spec/backend/runtime-gateway.md:140`。
- Extension runtime action 要求 `RuntimeContext::Session` 携带 `project_id`，backend placement 通过 `RuntimeTarget::Backend` 由宿主侧组装；见 `.trellis/spec/backend/runtime-gateway.md:150`、`.trellis/spec/backend/runtime-gateway.md:151`、`.trellis/spec/backend/runtime-gateway.md:152`。

### Code Patterns

#### Current Surface Facade

- `AgentRunRuntimeSurfaceQueryPort` exposes `current_runtime_surface` and `current_runtime_surface_with_backend` as application current-surface entrypoints; evidence: `crates/agentdash-application/src/agent_run/runtime_surface.rs:56`-`68`.
- The query resolves `RuntimeSessionExecutionAnchor`, `LifecycleRun`, `LifecycleAgent`, current `AgentFrame`, typed capability, typed VFS, MCP servers and backend anchor in one closure; evidence: `crates/agentdash-application/src/agent_run/runtime_surface.rs:86`-`98`, `:164`-`:218`, `:221`-`:249`.
- Backend-required consumers call `current_runtime_surface_with_backend`, which converts missing anchor into typed `MissingRuntimeBackendAnchor`; evidence: `crates/agentdash-application/src/agent_run/runtime_surface.rs:264`-`282`, `:410`-`:427`.
- `AgentRunRuntimeSurface` is already DTO-like and does not expose the domain `AgentFrame`; it returns runtime address, surface frame id/revision, capability state, VFS, MCP servers, optional backend anchor, identity and provenance; evidence: `crates/agentdash-application/src/agent_run/runtime_surface.rs:286`-`303`.
- The older / lower-level current frame resolver still returns domain entities and is therefore unsuitable as RuntimeGateway/API current-surface facade; evidence: `crates/agentdash-application/src/lifecycle/session_association.rs:32`-`43`, `:53`-`:59`.

#### RuntimeGateway MCP

- `RuntimeSessionMcpAccess` is the backing port for `mcp.list_tools` and `mcp.call_tool`; evidence: `crates/agentdash-application/src/runtime_gateway/session_actions.rs:14`-`15`, `:65`-`:77`.
- `McpListToolsProvider` and `McpCallToolProvider` are `SessionRuntime` providers and only extract `session_id` from RuntimeContext before delegating to the access port; evidence: `crates/agentdash-application/src/runtime_gateway/session_actions.rs:94`-`157`, `:175`-`:227`.
- `CurrentSurfaceRuntimeMcpAccess` is query-backed: it calls `current_runtime_surface_with_backend(session_id, "runtime_mcp_tool_discovery")`, then builds `McpToolDiscoveryRequest` with same-source MCP servers, capability state, VFS, backend anchor and identity; evidence: `crates/agentdash-application/src/runtime_gateway/mcp_access.rs:23`-`55`, `:92`-`:108`.
- `mcp.call_tool` re-discovers current entries and only executes a matching currently visible tool; evidence: `crates/agentdash-application/src/runtime_gateway/mcp_access.rs:72`-`88`, `:122`-`:132`.
- Existing tests cover idle MCP list using runtime-surface backend anchor, tool matching, hidden capability filtering, provider validation and missing tool mapping; evidence: `crates/agentdash-application/src/runtime_gateway/mcp_access.rs:455`-`505`, `crates/agentdash-application/src/runtime_gateway/session_actions.rs:368`-`459`.

#### RuntimeGateway Actor / Context Admission

- `RuntimeActor` session actors are `AgentSession`, `UserCanvas`, `WorkflowNode`, `SessionUser`; `PlatformUser` / `EnvironmentSetup` are setup actors; evidence: `crates/agentdash-application/src/runtime_gateway/types.rs:94`-`144`.
- `RuntimeContext::Session` carries `session_id`, optional `project_id`, optional `workspace_id`; evidence: `crates/agentdash-application/src/runtime_gateway/types.rs:146`-`180`.
- `surface_for_actor` and `invoke` both validate actor/context shape; for Session actions they only enforce non-empty context session id, session-bound actor and matching actor/context session id; evidence: `crates/agentdash-application/src/runtime_gateway/gateway.rs:65`-`80`, `:83`-`:127`, `:162`-`:193`.
- Existing tests cover actor-aware surface filtering and mismatched session rejection; evidence: `crates/agentdash-application/src/runtime_gateway/gateway.rs:396`-`450`.

## Caveats / Not Found

- `python ./.trellis/scripts/task.py current --source` returned no active task. The user provided the exact task directory and exact output file, so this report uses that explicit path.
- No business source code was modified; only this research file was updated.
- No tests were run; test names above are evidence from source inspection and proposed regression coverage.
- `resolve_session_frame_vfs` and `get_current_runtime_backend_anchor` do not exist as current symbols in the inspected crates. The active implementation appears to have converged on `AgentRunRuntimeSurfaceQueryPort::{current_runtime_surface,current_runtime_surface_with_backend}`.
- Critical release-risk caveat: Canvas runtime invoke and bridge currently do not visibly validate that the provided `session_id` belongs to the same Project as the Canvas before Gateway invoke. Extension runtime does query the session current surface, but the route should still explicitly reject path Project / session Project mismatch before using the path Project for extension installation lookup.
