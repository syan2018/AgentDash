## Findings

1. `AgentFrame` canonical surface is currently a multi-column revision row, not a single canonical typed document.
   `crates/agentdash-domain/src/workflow/agent_frame.rs:10` defines `AgentFrame`; the model-visible/runtime surface is split across `effective_capability_json`, `context_slice_json`, `vfs_surface_json`, `mcp_surface_json`, `execution_profile_json`, `visible_canvas_mount_ids_json`, and `visible_workspace_module_refs_json` at `crates/agentdash-domain/src/workflow/agent_frame.rs:15`. This directly affects Q-006 because generated/read-only projection columns are not just projections today; they are the persisted write surface.

2. The typed handoff exists, but it reads/writes the same multi-column storage.
   `FrameSurfaceDraft` carries typed `capability_state`, `vfs`, `mcp_servers`, context bundle summary, and execution profile at `crates/agentdash-application-agentrun/src/agent_run/frame/surface.rs:41`. `AgentFrameSurfaceExt` reads typed values back from individual JSON columns at `crates/agentdash-application-agentrun/src/agent_run/frame/surface.rs:71`. `AgentFrameBuilder::with_capability_state` splits one `CapabilityState` into capability / VFS / MCP columns at `crates/agentdash-application-agentrun/src/agent_run/frame/builder.rs:136`; `with_vfs_typed` and `with_mcp_servers` can also set VFS/MCP separately at `crates/agentdash-application-agentrun/src/agent_run/frame/builder.rs:148` and `crates/agentdash-application-agentrun/src/agent_run/frame/builder.rs:157`. This affects D-011/Q-006 because the code has a typed builder boundary, but still permits multi-column overwrite composition rather than a single canonical surface document.

3. `FrameRuntimeSurface` is a projection from the split `AgentFrame` columns.
   `crates/agentdash-application-ports/src/frame_launch_envelope.rs:19` defines `FrameRuntimeSurface`, and `from_frame` copies `effective_capability_json`, `context_slice_json`, `vfs_surface_json`, and `mcp_surface_json` into separate values at `crates/agentdash-application-ports/src/frame_launch_envelope.rs:31`. `build_envelope_from_frame` then starts from `FrameRuntimeSurface::from_frame` at `crates/agentdash-application/src/frame_construction/mod.rs:418`, reads `FrameSurfaceDraft::from_frame` at `crates/agentdash-application/src/frame_construction/mod.rs:420`, and closes the launch surface at `crates/agentdash-application/src/frame_construction/mod.rs:480`. 4. Append-only revision writing exists through `AgentFrameBuilder`, but current revision construction carries forward visible refs.
   `AgentFrameBuilder::build` persists a new frame via `repo.create` at `crates/agentdash-application-agentrun/src/agent_run/frame/builder.rs:225`. `build_uncommitted` chooses next revision from `repo.get_current(agent_id)` at `crates/agentdash-application-agentrun/src/agent_run/frame/builder.rs:240`, then carries forward old capability/context/VFS/MCP/execution fields at `crates/agentdash-application-agentrun/src/agent_run/frame/builder.rs:248` and carries forward visible refs at `crates/agentdash-application-agentrun/src/agent_run/frame/builder.rs:273`. This affects D-011 because the append-only path is present, but "current" still means highest revision and visible refs are treated as accumulated revision state.

5. Historical visible refs can still be mutated in place through repository helpers.
   Domain helpers mutate an `AgentFrame` object in place at `crates/agentdash-domain/src/workflow/agent_frame.rs:86` and `crates/agentdash-domain/src/workflow/agent_frame.rs:113`. `AgentFrameRepository` exposes `append_visible_canvas_mount` and `append_visible_workspace_module_ref` at `crates/agentdash-domain/src/workflow/repository.rs:101`.
