# AgentDashboard - 核心设计框架

**状态：** 框架定义阶段  
**原则：** 只定义抽象和关系，不涉及具体实现

---

## 1. 核心概念

### 1.1 状态容器（State Container）

定义：承载数字生产全过程状态的抽象空间

**属性：**
- 唯一标识（ID）
- 状态快照（某一时刻的完整状态）
- 历史轨迹（状态变更记录）
- 关联上下文（设计文档、资源引用等）

**边界：**
- 不定义存储介质（本地/远端/混合）
- 不定义隔离机制（分支/快照/其他）
- 不定义容量限制

### 1.2 状态迁移（State Transition）

定义：状态从一种形态转变为另一种形态的规则

**组成：**
- 源状态（前置条件）
- 目标状态（后置结果）
- 迁移规则（触发条件与验证逻辑）
- 执行者（谁/什么来执行迁移）

**类型：**
- 自动迁移：满足条件即触发
- 人工迁移：需人工确认后触发
- 半自动迁移：自动触发但人工验收

### 1.3 故事（Story）

定义：**用户价值单元**，承载用户视角的工作需求，负责维护完整上下文并编排执行

**结构：**
- 任务列表（Tasks）- 下属执行单元引用（核心关系，一对多）
- 上下文（Context）- 设计信息、资源引用、规范文档等
- 验收标准（Acceptance Criteria）- 完成定义
- 状态（当前所处状态）

**组织方式（用户自定义）：**
- Story之间的层级关系是**视图层的组织概念**，非核心状态
- 用户可自定义分组：按项目、模块、时间等维度
- 组织方式不影响执行流程（Agent只关心Task）
- 系统提供**可选的视图**供用户组织Story，但不强制

**职责：**
- 从用户角度描述需求和目标
- 接收并维护设计信息注入
- 拆解工作流为可执行任务
- 编排任务的执行顺序和依赖
- 聚合任务结果并验证验收
- **不直接绑定agent进程**

**生命周期：**
创建 → 上下文注入 → 拆解完成 → 编排执行 → 验收完成/失败

### 1.4 任务（Task）

定义：**执行单元**，绑定具体agent进程的容器，负责实际执行工作

**结构：**
- 所属故事（Parent Story）- 引用上级价值单元
- 执行环境（Environment）- 工作空间/隔离环境引用
- Agent绑定（Agent Binding）- 执行该任务的agent实例
- 输入上下文（Input Context）- 从故事继承+特定注入
- 输出产物（Artifacts）- 执行结果

**职责：**
- 在隔离环境中执行具体操作
- 维护agent进程生命周期
- 捕获执行状态和产物
- 向故事报告执行结果
- **一对一绑定agent进程**

**生命周期：**
待分配 → 已分配 → 执行中 → 待验证 → 完成/失败

**与故事的区别：**

| 维度 | 故事（Story） | 任务（Task） |
|------|------------|------------------|
| 定位 | 用户价值单元 | 执行容器 |
| 视角 | 用户视角（What） | 执行视角（How） |
| 核心关系 | 包含多个Task | 隶属于一个Story |
| Story间关系 | 用户自定义视图（非核心） | - |
| 结构 | 树状层次 | 扁平（隶属于故事） |
| Agent绑定 | 不绑定 | 一对一绑定 |
| 上下文 | 维护完整设计信息 | 接收派生上下文 |
| 职责 | 描述、拆解、编排、验收 | 执行、捕获、报告 |
| 实例化 | 由用户或PM创建 | 由故事拆解时创建 |
| 命名来源 | 敏捷开发标准术语 | 对齐Agent Task工具 |

## 2. 系统架构

### 2.1 三层结构

```
┌─────────────────────────────────────────────┐
│  编排层（Orchestration）                      │
│  - 任务拆解策略                               │
│  - 执行流程编排                               │
│  - 人机协作模式                               │
└──────────────────┬──────────────────────────┘
                   │
┌──────────────────▼──────────────────────────┐
│  状态层（State Management）                  │
│  - 状态容器管理                               │
│  - 状态迁移控制                               │
│  - 验证规则执行                               │
└──────────────────┬──────────────────────────┘
                   │
┌──────────────────▼──────────────────────────┐
│  连接层（Connectivity）                       │
│  - 工作空间连接                               │
│  - 多后端管理                                 │
│  - 用户会话管理                               │
└─────────────────────────────────────────────┘
```

### 2.2 多对多连接模型

**用户 ↔ 后端：**
- 一个用户可连接多个后端（本地/远程/多项目）
- 一个后端可服务多个用户（共享工作空间）

**后端 ↔ 工作空间：**
- 后端维护一个或多个工作空间
- 工作空间是状态容器的物理/逻辑载体

## 3. 关键设计原则

### 3.1 状态即真相
- 所有操作都体现为状态变更
- 状态快照可回溯、可对比
- 历史轨迹完整记录

### 3.2 验证可插拔
- 验证逻辑与迁移逻辑解耦
- 支持多种形式（Agent审查/脚本检查/人工确认）
- 验证规则可配置、可扩展

### 3.3 编排无侵入
- 编排策略不侵入状态层
- 支持多种编排模式（自动化链、人工审批、混合模式）
- 策略可切换、可组合

### 3.4 连接透明化
- 用户无感知切换本地/远程后端
- 工作空间位置对上层透明
- 连接管理独立演化

## 5. 使用场景验证

### 场景A：游戏项目故事产出
- 设计文档注入 → 故事接收完整上下文
- 故事拆解 → 创建任务（每个任务绑定特定agent）
- 分发执行 → 任务状态迁移到"执行中"（agent进程启动）
- 结果验证 → 触发验证规则（任务完成报告）
- 聚合验收 → 故事收集任务结果后完成验收

### 场景B：多项目并行管理
- 用户连接后端A（本地P4项目）
- 用户连接后端B（远程服务器项目）
- 每个后端维护独立状态容器
- 用户在统一界面管理多项目故事
