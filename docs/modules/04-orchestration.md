# 模块：编排引擎（Orchestration）

## 核心思想

编排是**可插拔的策略**，不是固定框架。

我们不预设"最佳编排方式"，因为：
- Agent能力在快速迭代，今天的最佳实践明天可能过时
- 不同场景需要不同策略（代码开发vs游戏设计vs内容生产）
- 用户应该能自由选择、切换、甚至自定义编排策略

系统只提供**不变的锚点**：
- Story需要被拆解为Task
- Task有依赖关系
- Task执行有生命周期
- 执行结果需要验证

"如何编排"交给可插拔的策略决定。

## 职责

- 加载并应用编排策略
- 根据策略决定Task执行顺序
- 协调Task的并行/串行执行
- 处理执行异常和重试
- 聚合结果推进Story状态

## 核心抽象

### 编排策略接口

```
OrchestrationStrategy {
  // 策略名称和描述
  name: string
  description: string
  
  // 核心方法：给定Story，返回执行计划
  createPlan(story): ExecutionPlan
  
  // 根据执行反馈调整计划（可选）
  adaptPlan(plan, feedback): ExecutionPlan
}
```

### 执行计划

```
ExecutionPlan {
  storyId: string
  tasks: Task[]           // 要执行的Task
  dependencies: Map       // Task间依赖关系
  parallelGroups: Group[] // 可并行执行的Task组
}
```

## 编排策略示例

策略是**可替换的组件**，示例包括：

**1. 规则驱动策略**
- 基于预设规则（顺序/并行/条件）
- 适合标准化流程

**2. Agent PM策略**
- PM Agent自主拆解和调度
- PM Agent根据执行反馈动态调整
- 适合复杂、不确定性高的任务

**3. 用户手动策略**
- 用户完全控制Task执行顺序
- 适合需要精细控制的场景

**4. 混合策略**
- PM Agent拆解，但关键步骤需人工审批
- 自动执行与人工介入结合

## 设计原则

1. **策略即插件**：新策略可无缝接入，不影响其他模块
2. **运行时切换**：Story可在不同策略间切换（如从自动改为手动）
3. **策略不侵入状态**：策略只影响"如何执行"，不修改Story/Task核心状态
4. **失败优雅**：策略异常时，回退到安全模式（如暂停等待人工）
