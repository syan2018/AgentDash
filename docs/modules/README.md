# 模块索引

系统被划分为以下子模块，每个模块独立演进，**策略可插拔**。

## 设计哲学

> **提供不变的锚点，不封死可能性**

我们不预设"最佳实践"，因为：
- Agent能力快速迭代，今天的方法明天可能过时
- 不同场景需要不同策略（代码/游戏/内容/...）
- 用户应该能自由替换、组合、自定义

系统只保证核心抽象稳定：
- Story和Task的关系
- 状态的完整性和可追溯性
- 模块间的接口契约

"如何实现"交给可插拔的策略。

## 核心模块

### 01. [连接管理（Connection）](./01-connection.md)
管理用户、客户端与后端的多对多连接。

**锚点**：必须支持同时连接多个后端（本地/远程）  
**策略可插拔**：连接协议、认证方式、重连策略

### 02. [状态管理（State）](./02-state.md)
核心存储层，维护Story、Task和变更历史。

**锚点**：状态必须完整、可追溯、一致  
**策略可插拔**：存储介质（文件/数据库）、序列化格式

### 03. [工作空间管理（Workspace）](./03-workspace.md)
管理Agent执行的隔离环境。

**锚点**：Task执行需要隔离环境  
**策略可插拔**：隔离方式（worktree/快照/容器/VM）、权限模型

### 04. [编排引擎（Orchestration）](./04-orchestration.md)
根据策略编排Story和Task的执行流程。

**锚点**：Story需要被拆解执行，Task有依赖关系  
**策略可插拔**：编排策略（规则驱动/Agent PM/手动/混合）

### 05. [执行调度（Execution）](./05-execution.md)
管理Task的实际执行。

**锚点**：Task需要绑定Agent执行，有生命周期  
**策略可插拔**：Agent类型、资源分配、超时策略

## 支撑模块

### 06. [信息注入（Injection）](./06-injection.md)
将设计信息、上下文注入到Story和Task。

**锚点**：执行需要上下文支撑  
**策略可插拔**：注入源类型、注入时机、合并策略

### 07. [验证机制（Validation）](./07-validation.md)
对执行结果进行验证。

**锚点**：结果需要验证  
**策略可插拔**：验证方式（Agent审查/脚本/人工）、通过标准

### 08. [视图组织（View）](./08-view.md)
提供Story的用户自定义组织方式。

**锚点**：用户需要组织和查看Story  
**策略可插拔**：视图类型（列表/树状/看板）、分组维度

### 09. [中继层（Relay）](./09-relay.md)
管理云端与本机后端之间的 WebSocket 通信。

**锚点**：本机必须能主动连接云端，云端必须能路由命令到任意在线本机  
**策略可插拔**：鉴权方式、消息序列化格式、重连策略、缓存策略

> 详细消息协议见 [relay-protocol.md](../relay-protocol.md)

## 模块设计原则

1. **策略可插拔**：每个模块的核心能力都可通过策略替换
2. **接口稳定**：模块间通过稳定的接口交互，实现细节可换
3. **失败优雅**：策略异常时，回退到安全模式
4. **不预设最佳**：提供多种选择，让用户根据场景决定
